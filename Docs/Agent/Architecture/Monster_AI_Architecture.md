# 怪物 AI 架构文档

最后更新：2026-05-10

## 1. 架构目标

本工程的 AI 方向改为《怪物猎人：世界》式大型怪物战斗 AI。目标不是把行为写死在控制器或角色 Tick 中，而是建立一套可由行为树驱动、由数据资产调参、由组件承载运行时状态的怪物 AI 骨架。

核心能力包括：

- 巡逻、发现玩家、接近、转向、攻击、后撤、吼叫、倒地、愤怒、换区、濒死等行为状态。
- 基于距离、角度、冷却、阶段、玩家数量、仇恨值和硬直状态的招式评分。
- 多人目标选择和仇恨切换。
- 部位耐久、部位破坏和硬直阈值。
- 行为树、黑板、EQS、GameplayTag 和评分系统的组合决策。
- 每次决策输出可观察调试信息，并写入上下文记忆。

## 2. 运行时分层

| 层级 | 主要产物 | 职责 |
| --- | --- | --- |
| 数据层 | `UMonsterAIDataAsset`、招式数据、部位数据 | 保存策划可调参数，禁止把招式条件硬编码到流程里。 |
| 状态层 | `UMonsterAIComponent` | 保存当前状态、阶段标签、目标、仇恨列表、黑板摘要、EQS 摘要和调试快照。 |
| 战斗层 | `UMonsterCombatComponent` | 评估招式、记录评分结果、检查可执行条件、维护冷却和失败原因。 |
| 移动层 | `UMonsterMoveComponent` | 输出接近、转向、后撤、等待等移动意图，供行为树或蓝图执行。 |
| 部位层 | `UMonsterPartComponent` | 管理部位耐久、破坏、硬直累计和倒地触发。 |
| 角色承载 | `AMonsterCharacter` | 只挂载组件，不在角色 Tick 中写怪物行为。 |
| 行为树层 | 行为树资产、黑板资产、EQS 查询 | 控制大流程，调用组件函数，按失败原因回退到移动、转向或等待。 |

## 3. 核心战斗循环

1. 行为树检查怪物是否死亡、濒死、倒地、硬直或换区。
2. `UMonsterAIComponent` 根据感知结果和仇恨表选择当前目标。
3. 行为树读取黑板距离、角度、目标数量、阶段标签和硬直状态。
4. `UMonsterCombatComponent` 使用数据资产中的招式表进行评分。
5. 如果最高分招式通过可执行条件，行为树进入执行攻击节点。
6. 如果招式失败，组件记录失败原因，行为树回退到接近、转向、后撤或等待。
7. 部位受到伤害时，`UMonsterPartComponent` 更新耐久、破坏状态和硬直累计。
8. 生命值或标签条件触发时，`UMonsterAIComponent` 切换阶段，例如愤怒或濒死。
9. 每轮关键决策写入调试快照和上下文记忆。

| 节点 | 目标 | 输入 | 输出 | 验证方式 | 失败处理 |

| 设计分析 | 明确怪物行为状态和循环 | 新需求文档、现有工程 | 架构文档、状态关系 | 文档完整性检查 | 先补文档，不进入代码 |

| 调试验证 | 提供可观察信息 | 评分、黑板、EQS、目标 | 调试快照和日志 | 构建、运行时日志 | 记录未验证风险 |

## 5. 长期架构决策

- 怪物行为不写在 `AIController` 或 `Character Tick` 中。
- 所有策划可调参数进入 `DataAsset` 或后续 `DataTable`。
- 招式选择使用评分系统，不使用大量 `if-else` 堆叠。
- 行为树只控制流程，具体状态和评分由组件提供。
- `GameplayTag` 用于描述状态、阶段、招式限制和部位状态。
- 调试输出必须包含当前状态、目标、招式、评分结果、黑板摘要、EQS 摘要和失败原因。
